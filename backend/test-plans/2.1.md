# Plano de Testes — 2.1: Implementar registro, login por e-mail/senha e recuperação de acesso

## Requisitos cobertos
- 11.1: Registro com validação de senha (mín. 8 chars, letras e números) e hash bcrypt
- 11.3: Senha validada com regex antes de hash
- 11.4: E-mail de verificação com token one-time; link de reset válido por 1h
- 11.5: Bloqueio de conta após 5 tentativas falhas por 15 min com notificação

## 1. Funcionalidade
> Corretude dos comportamentos especificados nos requisitos.

Relevante: SIM

### Partições de equivalência identificadas
| ID | Partição | Tipo | Comportamento esperado |
|----|----------|------|------------------------|
| P1 | E-mail único + senha válida | válida | `register()` → cria profissional, envia e-mail de verificação |
| P2 | E-mail já cadastrado | inválida | `register()` → CONFLICT |

| P7 | Senha incorreta | inválida | `login()` → UNAUTHORIZED, incrementa failedLoginAttempts |
| P8 | Conta bloqueada (lockedUntil > now) | inválida | `login()` → RATE_LIMITED com retryAfterSeconds |
| P9 | E-mail não cadastrado | inválida | `login()` → UNAUTHORIZED (sem revelar inexistência) |
| P10 | 5ª tentativa falha | edge | Bloqueia conta + notifica profissional |
| P11 | Token de verificação válido | válida | `verifyEmail()` → marca emailVerifiedAt |

| P20 | Logout com refresh token válido | válida | `logout()` → deleta token do store |

## 2. Confiabilidade
> Comportamento sob falha, recuperação, tolerância a erros.

Relevante: SIM — bloqueio de conta e rotação de tokens são mecanismos de tolerância a abuso.

### Casos cobertos pelos testes de funcionalidade
- TC-F-12/TC-F-13: comportamento seguro após 5 falhas consecutivas
- TC-F-17: recuperação automática após expiração do bloqueio
- TC-F-27: rotação de refresh token previne replay attack
