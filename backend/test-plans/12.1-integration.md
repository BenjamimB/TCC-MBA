# Plano de Testes — 12.1: Testes de Integração V1

## Escopo
Cobre todas as tasks V1 implementadas: 1.1, 1.2, 1.3, 2.1, 3.1, 3.2, 4.1, 6.1–6.4, 7.1–7.3.

## Estratégia geral

### Infraestrutura real, adaptadores externos mockados
| Recurso | Estratégia |
|---------|------------|
| PostgreSQL | Real (Docker — `tcc-mba-postgres-1:5432`) |
| Redis | Real (Docker — `tcc-mba-redis-1:6379`) |
| AI Gateway (Maritaca / Claude) | Mock via `jest.fn()` injetado pelo módulo de teste |
| WhatsApp API (Meta) | Mock via `jest.fn()` injetado pelo módulo de teste |
| E-mail (notificações) | Mock via `jest.fn()` |

### Setup / teardown
- `beforeAll`: bootstrap `NestApplication` com `AppModule` real; iniciar transação de banco
- `afterEach`: truncar tabelas em ordem de dependência para isolar testes
- `afterAll`: fechar app, encerrar conexões Prisma e Redis

### Arquivo base
`test/integration/` — separado de `test/` (e2e) e `src/**/*.spec.ts` (unit)
Regex: `.*\\.integration\\.spec\\.ts$`
Runner: `--runInBand` (testes sequenciais por conta de Redis compartilhado)

## Suite 2 — Autenticação ponta a ponta com banco real
**Task:** 2.1 | **Arquivo:** `auth.integration.spec.ts`

### Objetivo
Verificar o ciclo completo de auth persiste e lê do banco corretamente; JWT e refresh token gerados são válidos.

### Casos de teste
| ID | Descrição | Pré-condição | Resultado esperado |
|----|-----------|--------------|-------------------|
| IT-A-01 | `POST /auth/register` cria profissional no banco | banco limpo | HTTP 201; linha criada em `professional` com `passwordHash ≠ senha` |
| IT-A-02 | Segundo registro com mesmo e-mail retorna CONFLICT | profissional já existe | HTTP 409 `{ code:"CONFLICT" }` |
| IT-A-03 | `POST /auth/login` com e-mail não verificado retorna UNAUTHORIZED | `emailVerifiedAt IS NULL` | HTTP 401 `{ code:"EMAIL_NOT_VERIFIED" }` |
| IT-A-04 | `POST /auth/login` com credenciais válidas retorna accessToken e refreshToken | conta verificada | HTTP 200; `accessToken` decodificável com `sub = professionalId`; refresh token salvo no banco |
| IT-A-05 | `POST /auth/login` com senha errada incrementa `failedLoginAttempts` | conta verificada | HTTP 401; `failedLoginAttempts = 1` no banco |
| IT-A-06 | 5 tentativas falhas consecutivas bloqueiam conta por 15 min | `failedLoginAttempts = 4` | HTTP 401 `{ code:"ACCOUNT_LOCKED" }`; `lockedUntil` preenchido |
| IT-A-07 | `POST /auth/refresh` retorna novo accessToken com refresh token válido | session ativa | HTTP 200; novo `accessToken` válido |
| IT-A-08 | `POST /auth/logout` invalida refresh token no banco | session ativa | HTTP 200; token removido do banco; segundo refresh retorna 401 |
| IT-A-09 | Endpoint protegido retorna 401 sem Bearer token | — | HTTP 401 |
| IT-A-10 | Endpoint protegido retorna 200 com Bearer token válido | token ativo | HTTP 200 |

**Verificação de contrato HTTP (bug BUG-02 do bug-report):**
> Todos os endpoints de auth devem ser testados via Supertest para garantir alinhamento entre rota, parâmetros e body shape.

## Suite 8 — Lock atômico Redis com concorrência real
**Task:** 7.2 | **Arquivo:** `slot-reservation.integration.spec.ts`

### Objetivo
Verificar que `SET NX` do Redis garante exclusividade real sob concorrência — o cenário que os testes unitários com mock não podem validar.

### Casos de teste
| ID | Descrição | Pré-condição | Resultado esperado |
|----|-----------|--------------|-------------------|
| IT-SR-01 | `reserve()` com Redis real persiste e expira o lock pelo TTL | Redis vazio | Lock existe; após TTL expira |
| IT-SR-02 | `release()` com Lua script remove apenas o lock do owner | lock ativo | Lock removido; sessão diferente não remove |
| **IT-SR-03** | **10 chamadas simultâneas a `reserve()` para o mesmo slot — exatamente 1 ok** | Redis vazio | `Promise.all` de 10: 1 resolve com `ok`, 9 resolvem com `SLOT_NOT_AVAILABLE` |
| IT-SR-04 | Lock expirado por TTL torna slot disponível novamente | TTL de 1s | Após 1s slot pode ser reservado por nova sessão |

> IT-SR-03 é o teste de concorrência prioritário definido na Task 12.2 — incluído aqui pois depende de Redis real, não de carga.

## Suite 10 — Fluxo de conversa ponta a ponta
**Tasks:** 6.3, 6.4 | **Arquivo:** `conversation.integration.spec.ts`

### Objetivo
Verificar a state machine do ConversationOrchestrator com estado real no Redis e persistência real no banco. AI e WhatsApp mockados.

### Mocks obrigatórios
- `AITriageService.detectIntent()` — retorna intenção configurável por teste
- `AITriageService.generateResponse()` — retorna string fixa
- `IWhatsAppGateway.sendTextMessage()` — captura mensagens enviadas

### Casos de teste
| ID | Descrição | Estado inicial | Resultado esperado |
|----|-----------|---------------|-------------------|
| IT-CO-01 | Mensagem de agendamento: IDLE → TRIAGING → BOOKING_COLLECTING | nenhuma sessão no Redis | Estado `BOOKING_COLLECTING` salvo no Redis; WhatsApp envia lista de slots |
| IT-CO-02 | Seleção de slot válido: BOOKING_COLLECTING → BOOKING_CONFIRMING | estado BOOKING_COLLECTING no Redis; slot disponível no banco | Lock Redis criado; estado `BOOKING_CONFIRMING`; confirmação enviada |
| IT-CO-03 | Confirmação: BOOKING_CONFIRMING → CONCLUDED | estado BOOKING_CONFIRMING + lock ativo | Appointment criado no banco; lock removido; estado `CONCLUDED` |
| IT-CO-04 | Timeout de inatividade encerra sessão e libera lock | estado BOOKING_CONFIRMING com TTL expirado | Redis sem chave de sessão; lock removido |
| IT-CO-05 | Segunda mensagem simultânea do mesmo paciente é bloqueada pelo lock de conversa | lock de conversa ativo | Segunda mensagem enfileirada ou descartada; estado não corrompido |
| IT-CO-06 | intent=unclear → mensagem de esclarecimento sem criação de appointment | AI retorna `unclear` | Nenhum appointment no banco; WhatsApp recebe mensagem de esclarecimento |
